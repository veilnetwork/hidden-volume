# Политика semver

[🇬🇧 English](../../en/reference/semver.md) · 🇷🇺 **Русский**

**Статус.** Pre-release: линия v0.x трактует каждый minor-bump
(0.x → 0.y) как потенциально ломающий. После релиза v1.0 эта
политика становится обязывающей.

Этот документ описывает, **что покрывает semver** для crate
`hidden-volume` и его соседа `hidden-volume-async` после v1.0.
Политика обязывающая: если изменение в будущем `1.y.z` её
нарушает — мы делаем YANK и перевыпускаем. Оглавление изменений
лежит в `CHANGELOG.md`; этот документ объясняет, *что считается*
ломающим изменением (breaking change).

## 1. Что покрывает semver

Три поверхности, для каждой — явные правила:

### 1.1 Публичный Rust API

Поверхность `pub` у `hidden-volume` и `hidden-volume-async` —
типы, traits, функции, модули, доступные из `lib.rs` — покрыта
semver. Конкретно:

- Удаление или переименование `pub`-элемента — major bump
  (`2.0.0`).
- Изменение сигнатуры существующей `pub`-функции таким образом,
  что ломается хотя бы один реалистичный caller, — major bump.
- Добавление нового `pub`-элемента — minor bump (`1.y.0`).
- Внутренние рефакторинги без влияния на `pub` — patch bump
  (`1.y.z`).
- Изменения только в документации — также patch bump.
- Baseline `cargo public-api` (планируется к v1.0) — источник
  истины.

**Что значит «реалистичный caller».** Caller, который
компилируется против последней выпущенной версии и пишет
идиоматичный Rust (без магии вида `..= total::pub_inner_macro!()`,
без трюков с форком приватных модулей). Граничные случаи,
требовавшие `unsafe` или импорта приватного через rustdoc-трюки,
не учитываются.

**Реализации trait для существующих типов.** Добавление нового
`impl Trait for PublicType` технически — minor bump, но на
практике может сломать downstream-код, опирающийся на вывод
типов (например, новая impl становится неоднозначной с их
собственной). Значимые новые impl мы документируем в секции
"Added" `CHANGELOG.md`, чтобы downstream мог планировать.

### 1.2 Формат на диске

Wire format (специфицирован в `docs/ru/reference/format.md`)
**заморожен на v1.0**. После этого:

- Библиотеки `1.x.y` ОБЯЗАНЫ корректно читать v1-контейнеры.
- Добавление нового значения `ChunkKind`, новой константы
  `Namespace` или новой возможности за reservation byte
  (§8 из `FORMAT_v1.md`) — minor bump, если оно
  read-tolerant для старых библиотек `1.x.y` (т. е. они
  отвергают новую возможность с задокументированной ошибкой,
  а не молча портят данные).
- Добавление возможности, которую старые `1.x.y` библиотеки НЕ
  МОГУТ толерантно обработать (молча неверно распарсят), — это
  поколение v2; см. `docs/ru/guide/migration.md`.
- Усиление параметров Argon2 (поднятие `Argon2Params::MIN`)
  помешает открывать старые контейнеры; это тоже major bump / v2.

Перекос сильный: предпочесть v2 v1.x-расширению, рискующему
молча сломать downstream.

### 1.2.1 Политика `#[non_exhaustive]`

Следующие pub-элементы помечены `#[non_exhaustive]`:

| Элемент | Вид | Почему |
|---|---|---|
| `Error` | enum | Вырос с 5 до 16 вариантов до 1.0 (audit pass 17 добавил `ContainerTooLarge` для write-side scan-budget gate); новые варианты ожидаются по мере появления операций. |
| `ChunkKind` | enum | Версия формата резервирует место под новые виды чанков; downstream `match` обязан содержать `_ =>`. |
| `PaddingPolicy` | enum | Новые политики (например, экспоненциальный рост) могут появиться в minor-релизах. |
| `IntegrityReport` | struct | Конструируется только библиотекой; будущие поля (тайминги, статистика дерева) могут добавляться. |
| `SpaceStats` | struct | То же — host-приложения только читают; библиотека может добавлять поля. |

Следующие pub-элементы намеренно НЕ `#[non_exhaustive]`:

| Элемент | Вид | Причина |
|---|---|---|
| `ContainerOptions` | struct | Конструирование через struct-expression — естественный API (`ContainerOptions { argon2: …, … }`). `#[non_exhaustive]` принудил бы использовать builder; мы согласны, что добавление поля — major bump после v1.0. |
| `RepackOptions` | struct | То же, что `ContainerOptions`. |
| `Argon2Params` | struct | Заморожен на уровне формата — добавление полей это поколение v2, а не расширение v1.x. |
| `Namespace` | newtype | `Namespace(pub u8)` — паттерн-матчится как `Namespace(byte)`; non_exhaustive это молча сломал бы. |
| Внутренние типы формата (`Header`, `Plaintext`, `Superblock`, `LeafNode`, `InternalNode`, `IndexNode`, `IndexRoot`, `CommitPayload`) | смешанные | Их конструирование — часть тестовой / parser_fuzz поверхности; добавление `#[non_exhaustive]` вынудило бы перелопатить тесты без выгоды для безопасности. Спецификация формата в `docs/ru/reference/format.md` — то, что закрепляет их: заморожены *байты*, а не in-memory shape struct. |

### 1.3 Cargo features

Features покрыты одним правилом: добавление новой feature —
minor bump (`1.y.0`), и новая feature ОБЯЗАНА быть аддитивной —
её перевод из отсутствия в присутствие не должен ломать
существующих callers. Удаление или переименование feature —
major bump.

Набор feature библиотеки на момент v1.0:

- `cli` — бинарь `hv` + dependency `clap`.
- `parallel-scan` — параллельный discovery scan на rayon
  (только Unix, opt-in, без `unsafe`).
- `mmap` — zero-copy путь сканирования через memmap2 (только
  Unix, opt-in `unsafe`).

Feature-флага `std` нет — крейт безусловно требует `std`
(`alloc`-only / `no_std` не входит в v1.0; async-обёртка, FFI surface
и Argon2id KDF — все требуют `std`).

У `hidden-volume-async` и `hidden-volume-ffi` на v1.0 нет Cargo
features — async всегда тянет tokio; FFI всегда тянет uniffi 0.31.
Внутренний крейт `hidden-volume-rt` тоже без features.

## 2. Что semver НЕ покрывает

Эти вещи могут меняться между любыми patch-версиями без
уведомления:

- **Внутренние модули.** Всё, что не `pub` из `lib.rs`.
- **Test-only типы и helpers.** Даже если они в `pub`-модулях,
  но в их docs написано «for tests only» / «for benches only».
- **Версии dependency.** `cargo update` после `1.x.y` может
  выдать другой `Cargo.lock`; покрываются только диапазоны
  версий из `Cargo.toml`.
- **MSRV компилятора.** Минимальная поддерживаемая версия Rust
  задокументирована в `README.md` и может подняться в любом
  minor-релизе. Цель — поддерживать последние 6 месяцев stable
  Rust в любой момент.
- **Числа в бенчах.** Цифры в
  `docs/ru/contributing/benchmarks.md` информационные;
  регрессия производительности — баг, который надо чинить, но
  не нарушение semver.
- **Тексты ошибок.** Варианты `Error::*` стабильны;
  human-readable строки в `Display` / `Debug` impls могут
  меняться.
- **Лог-строки.** Библиотека не логирует; downstream может
  логировать — а может и нет.
- **Количество тестов.** «У нас 41 файл тестов» — это
  документация, не контракт.
- **Внутренние комбинации perf-feature.** `parallel-scan` может
  переключить свой 4-thread cap, изменить конфиг rayon-пула и
  т. д. без semver-эффекта, пока контракт публичного API
  держится.

## 3. Соответствие версии библиотеки и поколения формата

До v1.0:

- Библиотеки `0.x` пишут формат, описанный в
  `DESIGN.md` / `FORMAT_v1.md`. Pre-1.0 формат **не** стабилен
  между bump-ами 0.x.y; не разворачивайте без чёткого плана
  миграции (см. секцию Status `README.md`).

После v1.0:

| Версия библиотеки | Поколение формата | Заметки |
|---|---|---|
| `1.x.y` | v1 (frozen) | Обратно совместимое чтение + запись внутри `1.*`. |
| `2.x.y` (гипотетически) | v1 read + v2 write | Однонаправленная миграция; см. `docs/ru/guide/migration.md`. |
| `3.x.y` (гипотетически) | только v2 | Чтение v1 удалено; пользователи обязаны мигрировать сначала через 2.x. |

Точная каденция deprecation (сколько minor-версий до удаления
чтения) — один цикл major-версии = минимум один календарный год
между появлением v2 и удалением чтения v1.

## 4. Политика yank

Мы сделаем YANK релиза, если:

- Обнаружен баг в криптографии или crash recovery, способный
  вызвать потерю данных или ослабление безопасности на уже
  отгруженном релизе.
- Отгружено нарушение semver (ломающее изменение в
  не-major-релизе).

Мы НЕ будем делать yank ради:

- Регрессий производительности (починим, без yank).
- Ошибок в документации (выпустим патч).
- Косметической неудовлетворённости дизайном API.

Yanked-релизы документируются в `CHANGELOG.md` секцией "Yanked"
с объяснением причины.

## 5. Pre-release-версии

`alpha` / `beta` / `rc` pre-release **явно** не покрываются
semver — они существуют для downstream integration testing и
могут быть отозваны или перебазированы без уведомления.
Используйте только если ваш downstream-потребитель занимает
позицию «экспериментально, заменяемо».

Сам релиз v1.0 следует такой последовательности (планово):

```
1.0.0-alpha.1       Внешний криптоаудит начинается здесь.
1.0.0-alpha.2..N    Правки по результатам аудита.
1.0.0-rc.1          Аудит завершён; формат заморожен.
1.0.0-rc.2..N       Last-mile bugfixes.
1.0.0               Релиз.
```

## 6. Гарантии вне semver

Помимо semver мы обязуемся:

- **Стабильность формата.** Контейнер v1, созданный `1.0.0`,
  ОБЯЗАН открываться идентично на `1.x.y` для любых `x`, `y`.
- **Покрытие тестами.** `cargo test --workspace --all-features`
  ОБЯЗАН проходить на анонсированной MSRV на каждом релизе.
- **Прослеживаемость аудита.** Каждое публичное изменение,
  затрагивающее криптографию, документируется в `CHANGELOG.md`
  и в соответствующем `docs/*_AUDIT.md` с датой и подписью
  ревьювера.
- **Обоснование ломающих изменений.** Major bump сопровождается
  файлом `BREAKING.md` (или секцией "Breaking" в `CHANGELOG.md`)
  с перечислением каждого слома, обоснованием и путём миграции.
