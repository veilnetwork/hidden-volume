# Контрибьютинг в hidden-volume

[🇬🇧 English](../../en/contributing/guide.md) · 🇷🇺 **Русский**

Спасибо за интерес к участию в проекте. Этот документ описывает
рабочий процесс разработки, стиль кода и ожидания при ревью.

## Быстрый старт

```sh
git clone <fork>
cd hidden-volume

# Build, test, lint — это то, что запускает CI.
cargo build --all-targets
cargo test --tests
cargo test --doc
cargo test --features async --tests
cargo clippy --all-targets -- -D warnings
cargo fmt --all -- --check
cargo doc --no-deps --lib
```

Если что-либо из этого падает до того, как вы внесли изменения,
пожалуйста, откройте issue — возможно, ваше окружение настроено
неправильно, либо это реальный регресс, о котором мы хотели бы знать.

## Стиль кода

- **Форматируйте через rustfmt** перед commit: `cargo fmt --all`.
  Файл `rustfmt.toml` проекта фиксирует те немногие оси, которые
  варьируются между контрибьюторами; остальное закрывают дефолты.
- **Чистый clippy**: `cargo clippy --all-targets -- -D warnings`.
  CI отклоняет предупреждения.
- **Избегайте эмодзи** в коде, комментариях, commit-message и описаниях PR.
- **Следуйте существующим паттернам**: просмотрите похожий соседний
  файл перед добавлением новой конструкции. У кодовой базы единая
  структура модулей (`mod.rs` re-exports, отдельные подфайлы для
  разных обязанностей) и единая обработка ошибок (возврат
  `crate::Result`, проброс через `?`).

## Обзор архитектуры

См. [`DESIGN.md`](../../../DESIGN.md) для формальной модели угроз и
on-disk формата. Высокоуровневая структура модулей:

```
src/
├── crypto/        — KDF, AEAD, BLAKE3 derivation, RNG, ct helpers
├── chunk/         — фиксированный формат chunk на 4096 байт
├── container/     — операции уровня файла, header, ContainerOptions
├── space/         — superblock на space, B+ tree IndexNode, лог DataBatch
├── tx/            — мульти-операционные транзакции с протоколом 3-fsync commit
├── padding/       — None | BucketGrowth | FixedRatio
├── open/          — discovery scan + recovery
├── async_api/     — feature-gated tokio-обёртка (под фичей `async`)
└── error.rs       — единое перечисление Error
```

## Какие изменения приветствуются

- **Исправления багов** — всегда. Включайте регресс-тест.
- **Документация** — README, rustdoc, DESIGN, документы аудита.
- **Тесты** — дополнительное покрытие property-тестами, edge cases,
  бенчмарки.
- **Производительность** — должна сопровождаться данными criterion-бенчмарков,
  показывающими выигрыш.
- **Совместимость** — дополнительные платформы, расширение CI-матрицы.

## Какие изменения требуют предварительного обсуждения

Откройте issue (или draft PR с префиксом `WIP:`) перед серьёзной
работой над:

- **Изменениями on-disk формата** — даже до 1.0 churn формата
  затрагивает каждую тестовую фикстуру и путь миграции. Мы готовы
  ломать совместимость по уважительной причине; просто сначала
  обсудим.
- **Заменами криптографических примитивов** — XChaCha20-Poly1305,
  Argon2id, BLAKE3 — это сознательный выбор, задокументированный
  в `DESIGN.md` §10.
- **Новыми внешними зависимостями** — каждая зависимость это
  риск supply-chain. Мы предпочитаем минимальный срез. Получите
  одобрение перед добавлением.
- **Ломающими изменениями API до 1.0** — допустимы в принципе
  (линия v0.x нестабильна), но координируйте, чтобы объединить
  несколько ломающих изменений в один релиз вместо churn.
- **Кодом `unsafe`** — сейчас в крейте его ноль. Добавление
  `unsafe` требует письменного обоснования (в описании PR и
  блоке комментариев в месте вызова).

## Стандарты тестирования

- **Каждое изменение кода имеет минимум один тест.** Либо новый
  тест, либо обновлённый существующий.
- **Property-тесты** для любой логики с нетривиальной машиной
  состояний или roundtrip формата — см. `tests/property_full.rs`
  и `tests/parser_fuzz.rs` как образец.
- **Crash injection** для любых изменений протокола commit или
  пути durability — расширяйте `tests/crash_recovery.rs`.
- **Никаких flaky-тестов.** Детерминизм важнее скорости. Если
  тест зависит от тайминга, замените его на стабильный эквивалент.

## Политика безопасности

См. [`SECURITY.md`](../../../SECURITY.md) для отчётов об уязвимостях.
Пожалуйста, НЕ открывайте публичный issue или PR для находок
по безопасности — используйте приватные каналы раскрытия,
указанные там.

Каталог `docs/` содержит записи аудитов:

- [`docs/en/security/audits/memory.md`](../security/audits/memory.md) — обращение с ключевым материалом
- [`docs/en/security/audits/constant-time.md`](../security/audits/constant-time.md) — анализ constant-time
- [`docs/en/security/audits/fsync.md`](../security/audits/fsync.md) — протокол durability

Если ваше изменение затрагивает `crypto/`, `space/` или `container/`
способом, влияющим на эти аудиты, обновите журнал аудита и
запросите ревьюера с криптографическим бэкграундом.

## Commit-message

Проект использует описательные многоабзацные commit-message —
см. `git log` для образца. Кратко:

- Subject в повелительном наклонении, ≤ 70 символов.
- Тело объясняет что + почему, перенос на ~72 символах.
- Ссылайтесь на номера соответствующих issue/PR, если применимо.
- Для существенных изменений включайте краткую сводку тестов
  в конце: "X/Y тестов зелёных; clippy чистый; cargo fmt чистый."

## Pull requests

- Открывайте против `master`.
- Одно логическое изменение на PR. Если у вас несколько —
  оформляйте отдельно: проще ревьюить, проще откатить при
  необходимости.
- Включайте "Test plan" в описание (какие сценарии вы прогнали,
  какие edge cases рассмотрели).
- Дождитесь прохождения CI перед запросом ревью.

## Лицензия

Контрибьюции имеют двойную лицензию MIT OR Apache-2.0,
совпадающую с проектом. Отправляя PR, вы соглашаетесь, что
ваша контрибьюция может распространяться под любой из лицензий.
