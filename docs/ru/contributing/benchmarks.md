# Performance baseline

[🇬🇧 English](../../en/contributing/benchmarks.md) · 🇷🇺 **Русский**

Запуск: `cargo bench --bench throughput`

Все цифры в этом документе получены за один прогон на x86_64 на
машине разработки (12 потоков, 64 GB RAM, Linux). Mobile ARM-железо,
как ожидается, в 2-3× медленнее на путях, доминируемых Argon2,
и в 1.5-2× медленнее на путях chunk I/O.

## Базовые цифры (закоммичены: этот commit)

Все бенчмарки используют [`Argon2Params::MIN`] (m=8 MiB, t=2, p=1) —
примерно самую слабую приемлемую конфигурацию. Production с
`Argon2Params::DEFAULT` (64 MiB / 3 итерации) занимает примерно
в 4× больше времени для любого пути, проходящего через KDF
(`create_space`, `open_space`, repack).

| Бенчмарк | Время (медиана) | Примечания |
|---|---:|---|
| `create_space` | **5.5 ms** | Argon2id MIN + начальные записи Superblock |
| `open_space` | **5.4 ms** | Argon2id MIN + O(N) scan + auto-vacuum |
| `commit_single_kv` | **5.8 ms** | Open + put + commit + close (доминирует Argon2) |
| `commit_100_kv` | **5.8 ms** | 100 put в одном Tx — тот же baseline, что и 1 put |
| `commit_1000_kv` | **6.1 ms** | Принуждает к B+ tree split — всего +5% над single |
| `commit_log_100` | **5.9 ms** | 100 записей лога в одном zstd-сжатом батче |
| `get_random_kv` | **36 µs** | KV-lookup, namespace на 1000 записей (2-уровневое B+) |
| `read_log` | **84 µs** | Lookup лога, 1000 msgs / 10 батчей (включая zstd decode) |
| `repack_1000` | **13 ms** | Полный repack 1000 KV + 100 записей лога |
| `open_large_sequential` | **52 ms** | 5 000 KV + 1 000 log + 10 000 garbage = ~10 K slot / 40 MiB контейнер |
| `open_large_parallel` | **18 ms** | Тот же контейнер, фича `parallel-scan` — **в 2.8× быстрее** |
| `open_50k_sequential` | **608 ms** | 50 K slot / ~200 MiB история размера messenger |
| `open_50k_parallel` | **264 ms** | То же, `parallel-scan` — **в 2.3× быстрее** |
| `open_100k_sequential` | **1499 ms** | 100 K slot / ~400 MiB messenger с тяжёлой историей |
| `open_100k_parallel` | **204 ms** | То же, `parallel-scan` — **в ~7× быстрее** (page-cache + sweet spot 4 потока) |
| `iter_log_full` | **484 µs** | Все 1 000 записей из лога на 5 батчей (legacy `iter_log`) |
| `iter_log_before_50` | **87 µs** | Самые свежие 50 записей (примитив пагинации messenger) |
| `verify_integrity` | **125 µs** | Полный обход Merkle по 2 namespace (1 000 + 100 KV) |

## Пагинация — это правильный путь

`iter_log_before_50` **в 5.6× быстрее**, чем `iter_log_full`,
несмотря на одинаковый базовый лог:

```
iter_log_full         484 µs   ← декодирует все 1 000 записей
iter_log_before_50     87 µs   ← декодирует только новейшие 50
```

Выигрыш масштабируется линейно с размером namespace: лог на
100 K сообщений всё равно займёт ~100 µs для первой обратной
страницы, тогда как `iter_log_full` пересечёт 50 ms и станет
заметным пользователю. Используйте `iter_log_after` /
`iter_log_before` для любого namespace, который растёт
неограниченно — см. `docs/en/guide/integration.md` §5.

## verify_integrity дёшев

Полный обход Merkle по нетривиальному дереву (1 100 KV-записей
в 2 namespaces) занимает 125 µs. Запуск его как self-test при
каждом старте приложения стоит фактически ничего. Рекомендуемая
частота: после sync с peer, после crash recovery host-приложения,
периодически как defense-in-depth аудит.

## Инсайты

### 1. Пол в 3 fsync

Любой commit занимает минимум ~5 ms из-за трёх барьеров fsync
(`Data → fsync → Commit → fsync → Superblock → fsync`). На современных
SSD это доминирующая стоимость; собственно вычисления (AEAD,
кодирование, zstd) занимают суб-миллисекунду.

Это значит, что **батчинг записей фактически бесплатен в пределах
лимита capacity chunk**. Tx, кладущий 1 запись, обходится в то же
wall-clock, что и Tx, кладущий 100 записей — оба платят 3 fsync.

Следствие для UX messenger: батчите исходящие сообщения
оппортунистически (короткое окно flush, например 50 ms), чтобы
снизить стоимость commit на сообщение.

### 2. B+ tree split дёшев

`commit_1000_kv` всего на 5% медленнее `commit_single_kv`,
несмотря на:
- pack_into_leaves с greedy first-fit на тысячах байт
- множество кодирований chunk Leaf + AEAD seals
- один Internal-узел сверху

Это валидирует выбор дизайна 2-уровневого B+ tree — стоимость
split хорошо амортизируется барьером 3 fsync.

### 3. Пути чтения очень быстрые

- KV-lookup: 36 µs (один обход дерева + AEAD-decrypt + бинарный поиск)
- Log-lookup: 84 µs (KV-lookup + zstd-разжатие + линейный скан в батче)

Даже на медленном mobile-железе это будет суб-миллисекунда.
Отзывчивость UI не упирается в storage.

### 4. Repack быстр

12 ms на repack 1000 KV + 100 сообщений лога. Для типичной
нагрузки messenger (~5000 контактов + 50000 сообщений) repack
занял бы примерно ~100-500 ms — допустимо делать как фоновую
задачу "cleanup" при старте приложения или после удаления.

## Тюнинг parallel-scan

Фича `parallel-scan` (на основе rayon) имеет три рычага тюнинга,
каждый из которых по отдельности был необходим, чтобы получить
реальное ускорение:

```
open_large_sequential                52 ms  (baseline)

→ par_iter().map().collect()        265 ms  ✗ гигантский промежуточный Vec, contention аллокатора
→ try_fold/try_reduce               265 ms  ✗ без изменений — это было не bottleneck
→ + крупнозернистый chunking (256)  141 ms  ✗ лучше, но всё ещё > sequential
→ + cap до 4 потоков (было 12)       47 ms  ✓ чуть ниже sequential
→ + кэшированный static pool         18 ms  ✓ ускорение в 2.8×
```

**Финальная имплементация.**
1. **Крупнозернистый chunking.** Каждый параллельный work-item
   обрабатывает 256 последовательных слотов sequentially (без
   накладных расходов на per-slot scheduling). Работа на слот —
   ~5 µs — намного ниже накладных расходов rayon на задачу,
   если не амортизирована.
2. **Cap на 4 потока.** AEAD-decrypt + small-chunk pread насыщают
   L1-cache и пропускную способность памяти задолго до того, как
   насыщают ядра. Эмпирическое масштабирование на 12-поточном x86
   dev-хосте:
   ```
   1 поток      51 ms  (sequential через rayon = baseline)
   2 потока     32 ms  (ускорение в 1.6×)
   4 потока     47 ms  (variance растёт; около baseline)
   12 потоков  141 ms  (в 3× МЕДЛЕННЕЕ — обрыв из-за contention)
   ```
   Мы делаем `min(4, available_parallelism)`, чтобы оставаться
   на правильной стороне обрыва независимо от количества ядер хоста.
3. **Кэш static pool.** Построение свежего `rayon::ThreadPool`
   на каждый вызов `open_space_parallel` стоит несколько ms и
   доминирует wall-clock для быстрых сканов. Pool строится один
   раз через `OnceLock` и переиспользуется между open.

**Когда включать.**
- ✓ Многоядерные хосты (≥4 логических) с нетривиальным размером
  контейнера (≥ ~10 K слотов / ~40 MiB). На dev-машине: open
  в 2.8× быстрее.
- ✗ Одноядерные mobile (Cortex-A53 класса). Capped pool
  схлопывается до 1 потока, и вы платите ~6 MiB размера бинарника
  rayon без ускорения. Оставьте фичу ВЫКЛ.
- ? Крошечные контейнеры (< 1 K слотов). Margin ускорения
  сжимается ниже пола накладных расходов rayon; не измерялось.
  Sequential подойдёт.

### Масштабирование

End-to-end open (`Container::open` + `open_space*` + `vacuum_orphans`)
на том же 12-поточном x86 dev-хосте по разным размерам контейнера:

| Контейнер | Sequential | Parallel | Ускорение | Throughput (par) |
|---|---:|---:|---:|---:|
| 10 K slot / 40 MiB | 52 ms | 18 ms | 2.8× | 2.2 GiB/s |
| 50 K slot / 200 MiB | 608 ms | 264 ms | 2.3× | 760 MiB/s |
| 100 K slot / 400 MiB | 1499 ms | 204 ms | 7.4× | 2.0 GiB/s |

Результат на 50 K — это **провал в кривой** — на 200 MiB
sequential read-ahead ещё работает (768 MiB/s), и 4-поточный
parallel получает только 2.3× ускорение. К 400 MiB sequential-путь,
по-видимому, выпадает из горячего пути page-cache (270 MiB/s, в 3×
медленнее на байт, чем при 10 K), тогда как parallel pread с
многих потоков продолжает агрессивно префетчить и держится
на ~2 GiB/s. Поэтому parallel-scan помогает **больше всего**
именно там, где это важнее всего: большие истории messenger
на многоядерных хостах.

Замечание о variance: разброс параллельной выборки на 100 K
был [162, 204, 274] ms (10 samples) — шире, чем на меньших
размерах, но медиана уверенно ниже диапазона sequential
[1367, 1499, 1627] ms. Даже худшая параллельная выборка
обходит лучшую sequential в 5×.

### Влияние UX для разработчиков messenger

Перевод цифр в UX-стоимость на разблокировку, видимую пользователю:

| Размер профиля пользователя | Sequential unlock | Parallel unlock |
|---|---:|---:|
| Лёгкий пользователь (~40 MiB) | 52 ms — невидимо | 18 ms — невидимо |
| Средний пользователь (~200 MiB) | **0.6 s** — заметно | 0.26 s — невидимо |
| Тяжёлый пользователь (~400 MiB) | **1.5 s** — UX-стоимость | **0.2 s** — невидимо |

Как только история пользователя превышает ~200 MiB, sequential
unlock пробивает порог "пользователь замечает" (>300 ms — см.
[порог Доэрти](https://en.wikipedia.org/wiki/Mental_chronometry)).
На 400 MiB это явный момент "приложение зависло?".

**Рекомендация для разработчиков messenger:** включайте
`parallel-scan` для любого многоядерного хоста с реалистичной
для messenger историей. Отключайте на одноядерных mobile
(cap в 4 потока схлопывается в 1 — ускорения нет, ~6 MiB
размера бинарника rayon впустую). См. матрицу "Когда включать"
выше для полного дерева решений.

**Поведенческая гарантия.** `tests/parallel_scan.rs` (6 сценариев)
утверждает, что параллельный путь производит тот же наблюдаемый
`SpaceState`, что и sequential — тот же superblock, те же
owned_slots, тот же commit_history, тот же результат
verify_integrity.

## Как интерпретировать регрессы

Если будущий commit выталкивает любую из этих цифр вверх на >25%,
расследуйте. Конкретно:
- **Commit-бенчмарки >7.5 ms**: где-то лишний fsync, или добавлено
  дорогое per-chunk вычисление.
- **Read-бенчмарки >100 µs / >150 µs**: обход дерева добавил
  уровень, или per-leaf decode стал медленнее.
- **Repack >15 ms / 1000**: перечисление или переписывание
  стало медленнее.

## Рекомендации по тюнингу под железо

Для use case messenger host-приложение должно выбирать параметры
Argon2 на основе класса устройства (DESIGN §11.1):

| Класс устройства | Рекомендуемые параметры | Примерный open_space |
|---|---|---|
| Low-end ARM (Cortex-A53, 2017+) | `Argon2Params::LIGHT` | ~30 ms |
| Mid-range ARM (телефоны последних 5 лет) | `Argon2Params::DEFAULT` | ~100 ms |
| Desktop / server-класс x86 | `Argon2Params::HEAVY` | ~250 ms |

Цифры в этом документе предполагают `MIN` для целей
бенчмарков. Для каждого пресета умножайте пути, доминируемые
Argon2 (create_space, open_space, repack), на соответствующее
отношение:

- LIGHT (m=16 MiB, t=3): ~1.5× MIN
- DEFAULT (m=64 MiB, t=3): ~4× MIN
- HEAVY (m=256 MiB, t=4, p=4): ~10-15× MIN

## Валидация perf-target v0.6 (`TASKS.md` L538)

Майлстоун v0.6 стремился к:

| Цель | Аспирация | Измерено (dev host x86) | Статус |
|---|---:|---:|---|
| Throughput parallel scan | ≥ 5 GiB/s на x86 | 2.0–2.2 GiB/s | **Не достигнут** в ~2.5× |
| Throughput parallel scan | ≥ 1 GiB/s на ARM | не измерялось | **Не измерено** |
| Throughput append | ≥ 50 MB/s на mobile flash | напрямую не бенчмаркилось | **Не измерено** |
| Throughput repack | ≥ 100 MB/s на x86 | ~333 MiB/s¹ | **Достигнут** ✓ |

¹ `repack_1000` (12 ms, ~4 MiB живых данных) → 333 MiB/s. Большие
контейнеры отдельно не бенчмаркились, но per-byte стоимость
доминируется AEAD re-seal + zstd, оба стабильны по throughput.

### Почему scan ниже аспирации в 5 GiB/s

Путь scan ограничен **AEAD-decrypt + small-chunk pread**, и на
12-поточном x86 dev-хосте per-thread потолок выглядит как
~500–600 MiB/s (XChaCha20-Poly1305 ~1.5 GiB/s без I/O,
дросселируется I/O-bound cap на 4 потока из "Тюнинг
parallel-scan" выше). Достичь 5 GiB/s потребовало бы либо
(a) снять cap на 4 потока — что эмпирическая кривая показывает
триггерит обрывы из-за contention — либо (b) перейти от
per-chunk AEAD к streaming AEAD конструкции, что ломает
инвариант discoverability (каждый chunk должен trial-decrypt
под ключом любого space независимо). **Потолок 2 GiB/s присущ
формату**, а не gap оптимизации на уровне кода. Мы принимаем
это и пересматриваем цель вниз в следующем майлстоуне:

> **Пересмотренная цель (v1.0):** parallel scan ≥ 1.5 GiB/s
> на x86 с фичей `parallel-scan`; ≥ 300 MiB/s на ARM Cortex-A53.

### ARM не измерен

Sandbox CI-окружение для этих бенчмарков не имеет ARM-железа.
Валидация на реальных телефонах Cortex-A53 / A76 **отложена до v0.8**,
когда появится FFI-слой (`hidden-volume-ffi`), и у нас будет
deployable `.aar` для замеров на устройстве. До тех пор цифры
ARM экстраполируются из правила большого пальца в шапке документа
(в 2-3× медленнее на Argon2; в 1.5-2× медленнее на chunk I/O).

### Throughput append

Текущий набор бенчмарков измеряет **commit**, а не raw append.
Wall-clock commit доминируется барьером 3 fsync (~5 ms пол на
SSD, многие секунды на дешёвой eMMC). Цифра bytes-per-second
полностью зависит от размера батча Tx: Tx с 1 KV-парой платит те
же 3 fsync, что и с 1000 — поэтому "append throughput"
вводит в заблуждение в этом дизайне. Host-приложения должны
батчить исходящие записи (окно flush 50 ms достаточно, чтобы
амортизировать пол fsync). Цель 50 MB/s поэтому переформулирована
как Tx-батченая цель:

> **Пересмотренная цель (v1.0):** ≥ 50 MB/s sustained, когда
> host-приложение батчит в Tx-commit ≥ 100 KB. С Tx-commit по
> 64 KB каждый (~12 сообщений по 5 KB в одном Tx), пол fsync
> в 5 ms транслируется в 12.8 MB/s — что согласуется с
> доминированием latency mobile flash, а не throughput.

### Воспроизведение

```sh
cargo bench --bench throughput            # baseline
cargo bench --bench throughput --features parallel-scan  # для параллельных путей
```

Медианные значения записываются в
`target/criterion/<bench>/new/estimates.json`. Прогоняйте снова
после любого commit, затрагивающего `space::commit_tx`,
`open::scan_and_recover*`, `crypto::aead` или `crypto::derive`.
